Claude 审批服务器 — 4+1 架构视图

基于 Kruchten 4+1 视图模型，从 5 个维度完整描述系统架构

1
逻辑视图 (Logical View)
功能分解 · 模块职责
逻辑视图关注系统的功能分解。核心分为 5 层：用户层发起任务和审批；AI 客户端调用 MCP 工具；核心服务层包含请求管理、HTTP API、前端界面、安全模块；外部服务适配层抽象了 ngrok、推送、持久化等外部依赖；最终由手机浏览器完成用户交互。
2
开发视图 (Development View)
代码结构 · 文件依赖
开发视图展示代码组织和模块依赖。核心只有两个 JS 文件：mcp-server.js 通过 require() 导入 server.js，复用其全部功能。配置文件 (.mcp.json, config.env) 含敏感信息，通过 .gitignore 排除。数据文件 (.data/*.json) 也不入版本控制。
3
进程视图 (Process View)
运行时进程 · 并发 · 通信
进程视图描述运行时行为。整个系统运行在单个 Node.js 主进程中，内含 HTTP Server、stdin 解析器、定时器、SSE 连接池等。主进程会 spawn() 一个 ngrok 子进程用于隧道。关键并发点：阻塞等待循环 (waitForStatus) 通过 1 秒轮询 store 对象实现，所有模块共享内存中的 store 对象。
4
物理视图 (Physical View)
网络拓扑 · 部署
物理视图展示部署拓扑。核心节点是开发电脑（运行 Node.js + ngrok），通过 ngrok 隧道暴露到公网。推送链路：server → 喵提醒 API → 微信 → 手机通知。审批链路：手机浏览器 → ngrok 公网 URL → 隧道 → localhost:8765。同一 WiFi 下可绕过 ngrok 直连。
+
场景视图 (Scenarios)
关键用例 · 串联 4 个视图
场景视图用 3 个关键用例串联前 4 个视图。场景 1 展示完整审批链路（用户→Claude→MCP→推送→手机→回传）；场景 2 展示多轮对话如何复用 conversation_id 和 messages 数组；场景 3 展示自愈机制的触发与恢复流程。每个场景都横跨逻辑、开发、进程、物理四个维度。
